风险管理

识别的风险：

  1. 技术风险：LLM API 稳定性
  2. 风险描述： 免费 API 代理可能不稳定，GFW 可能封锁
  3. 应对方法：
    • 实现了多层代理配置（proxy_config.py）
    • 支持切换不同的 API 端点
    • 本地缓存和错误重试机制

  4. 效果： 在开发过程中遇到过 2 次代理失效，通过快速切换端点解决

  5. 用户体验风险：过度打扰

  6. 风险描述： 声线评论过于频繁可能打断写作流
  7. 应对方法：
    • 能量池系统：只有积累到 40 能量才触发
    • 碰撞检测：避免重复高亮同一段文字
    • 用户可以关闭特定声线

  8. 效果： 测试反馈显示 5 秒轮询 + 40 能量是较好的平衡点

  9. 数据迁移风险：localStorage → 数据库

  10. 风险描述： 用户可能在迁移过程中丢失数据
  11. 应对方法：
    • 实现了导入/导出功能
    • 首次登录时显示迁移向导
    • 保留 localStorage 作为备份

  12. 效果： 零数据丢失，迁移过程平滑

  13. 项目管理风险：Scope Creep

  14. 风险描述： 功能不断膨胀，可能无法按时交付
  15. 应对方法：
    • 严格的 MVP 思维：每个功能都问 "是否必须？"
    • Vibe-coding 方法：快速原型 → 观察 → 决定是否保留
    • 使用 TodoWrite 工具跟踪任务
  16. 效果： 虽然增加了 Deck System 等功能，但核心体验始终聚焦

从工程导师学到的经验

目前我们暂未请教学院的工程导师，以下是我们自己总结出的经验。

核心理念：

  1. "Code is cheap. Show me your talk."
  2. 在写代码前充分思考和设计
  3. 使用对话来理清思路

  4. 代码应该是思考的结果，而不是试错的过程

  5. 最小实现原则

  6. "我们在用复杂度去买功能"
  7. 每增加一个功能都要评估复杂度成本

  8. 简单可维护比完美但复杂的方案更好

  9. 基于事实的调试方法

  10. "Never use a 'match' based approach to debug. Use a 'print' based approach."
  11. 先观察现象，不要猜测

  12. console.log 和 print 是最可靠的工具

  13. 专业客观性

  14. 尊重事实比尊重自我更重要
  15. 当发现错误时，立即承认并修正
  16. "A cat dies every time you are overly agreeable"
  17. 保持平衡和坦率的沟通

项目管理特点

我们的方法：

  1. 非线性进展
  2. 不是每天固定距离前进
  3. 采用 探索-固化-重构 的节奏

  4. 有时一天完成核心功能，第二天重构，比如写作界面的 UI，时间线 UI，评论触发机制等都经历过重构。

  5. 优点：

  6. 高度灵活： 可以快速响应新想法（如 Deck System 是中途加入的）
  7. 质量优先： 不为了 deadline 而妥协代码质量
  8. AI 深度集成： 使用 Claude Code 作为主力开发工具，效率极高

  9. 文档即代码： CLAUDE.md 和代码同步更新，onboarding 成本低

  10. 缺点：

  11. 进度难以预测： 很难提前知道某个功能需要多久
  12. Scope 控制难： 容易陷入 "这个功能也很酷" 的陷阱
  13. 单人开发局限： 缺少 code review 和设计讨论
  14. 测试覆盖不足： 主要依赖手动测试，缺少自动化测试

进度模式分析：

我们的进度更像是 "波浪式前进" 而非匀速直线：

Week 1: 核心编辑器 + LLM 集成 (快速原型)
Week 2: 碰撞检测 + 状态管理 (深度打磨)
Week 3: 认证系统 + 数据库 (架构升级)
Week 4: Deck System + UI 优化 (功能扩展)

这种模式的好处是每个阶段都能交付可用的功能，而不是等到最后才集成。
